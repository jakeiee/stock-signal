## 产品概述

针对2个月CodeBuddy使用经验总结的5大核心问题，实施全套项目治理体系改进：

1. **上下文/规则/记忆/技能/代理输出不稳定** - 规则过载、记忆噪音、技能协作混乱
2. **重大事故频发** - 核心功能被修改至无法运行，缺乏安全机制
3. **数据源接入不稳定** - 频繁接入不稳定数据源或理解偏差
4. **问题识别准确性波动** - 上下文管理问题导致识别率下降
5. **缺乏项目实践方案** - 一直在修修补补，无系统化架构

## 核心功能

### 1. 工作空间治理

- 清理50+个时间戳目录（/Users/liuyi/WorkBuddy/）
- 实施保留策略（保留最近10个+加星标）
- 自动归档机制（90天前归档到cold storage）
- 配置自动化任务定期清理

### 2. 模块化架构升级

- 实现标准模块接口（BaseModule）
- 实现模块自注册机制（ModuleRegistry）
- 适配现有3个模块（dividend_monitor, market_monitor, portfolio）
- 支持后续新增模块自动发现和加载

### 3. 规则与记忆系统重构

- 规则分层架构（核心原则≤50行 + 领域规则 + 按需加载）
- 记忆精炼机制（自动提取黄金规则，删除重复记录）
- 规则自动提升（使用≥3次自动提升到上层）

### 4. 代码安全网

- Git分支策略（修改前自动创建时间戳分支）
- 修改影响分析工具（safe_edit.py）
- 自动回滚机制（测试失败时自动回滚）

### 5. 自动调整机制

- 数据源自动故障转移（主源失败≥3次自动切换备用源）
- 模块性能监控（执行时间超阈值自动告警）
- 规则自动提升机制

- 基于现有tools/harness/框架雏形进行升级（已探索确认存在完整实现）

### 5. 自动调整机制

#### 数据源自动故障转移

在 `data_sources/` 中添加装饰器：

```python
def auto_failover(primary_func, fallback_func):
    fail_count = 0
    def wrapper(*args, **kwargs):
        try:
            result = primary_func(*args, **kwargs)
            fail_count = 0
            return result
        except Exception as e:
            fail_count += 1
            if fail_count >= 3:
                log_error(f"主数据源连续失败{fail_count}次，自动切换")
                return fallback_func(*args, **kwargs)
            return fallback_func(*args, **kwargs)
    return wrapper
```

## 关键技术决策

| 决策点 | 选择 | 理由 |
| --- | --- | --- |
| 模块发现机制 | pkgutil.iter_modules | 标准库，无额外依赖，支持动态发现 |
| 规则加载策略 | 分层 + 强制核心 | 解决规则过载问题，确保核心规则执行率 |
| 记忆精炼频率 | 每周一次 | 平衡及时性和系统开销 |
| 数据源故障转移 | 装饰器模式 | 透明化，不侵入业务逻辑 |
| Git安全网 | 强制分支策略 | 防止重大事故的最有效手段 |
| 性能监控阈值 | 60秒 | 金融数据获取的合理上限 |


## 性能考虑

- **低开销**: 所有新工具均为纯Python实现，无额外依赖
- **惰性加载**: 模块和规则按需加载
- **增量处理**: 记忆精炼只处理新增记录
- **非阻塞**: 性能监控异步执行，不影响主流程
